01 The Challenge

A portfolio site is a public-facing representation of your engineering standards. Shipping broken links, spelling mistakes, or malformed Markdown directly to production — even on a personal project — undermines the credibility the site is meant to build.

The goal was to treat this site with the same discipline as a production system: no direct pushes to the deployment branch, automated validation on every change, and a fully hands-off deploy once quality gates pass.

Design Constraint
The pipeline had to be entirely free to operate — no paid CI runners, no external services. GitHub Actions on a public repository with GitHub Pages deployment satisfied all requirements at zero cost.
02 Approach

The pipeline uses two protected branches — main (authoring) and release (deployment source) — with a GitHub Actions workflow that gates the promotion between them.

1
Markdown lint — markdownlint-cli
Enforces consistent Markdown syntax across all content files: heading hierarchy, list formatting, trailing whitespace, line length. Configuration stored in .markdownlint.json at the repo root.
2
Spell check — cspell
Runs across all Markdown and HTML content. A custom cspell.json wordlist handles technical terms (DevOps tool names, CLI flags, proper nouns) that would otherwise trigger false positives.
3
HTML & link validation — htmlproofer
Builds the Hugo site first, then runs htmlproofer against the output directory. Validates internal links, checks external URLs, catches missing alt tags on images, and validates anchor references.
4
Auto-merge to release on pass
If all three checks pass, the workflow opens and merges a PR from main into release automatically using a scoped GITHUB_TOKEN. No human step required.
5
Hugo build & Pages deploy
A second workflow triggers on push to release, runs hugo --minify, and deploys the public/ output to GitHub Pages using the official actions/deploy-pages action.
# .github/workflows/validate.yml (abridged) on: push: branches: ['main'] jobs: validate: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - name: Markdown lint run: npx markdownlint-cli 'content/**/*.md' - name: Spell check run: npx cspell 'content/**' - name: Build site run: hugo --minify - name: HTML & link validation run: | gem install html-proofer htmlproofer ./public \ --disable-external=false \ --checks Links,Images,Scripts - name: Merge to release if: success() run: gh pr merge --auto --merge env: GH_TOKEN: ${{ "{{" }} secrets.GITHUB_TOKEN {{ "}}" }}
03 Issues Encountered
blocker cspell false positives on DevOps terminology
Tool names like htmlproofer, markdownlint, CLI flags, and Hugo shortcode syntax all flagged as misspellings. Resolved by building a project-level cspell.json with a custom words array and ignorePaths for generated files. Adding this incrementally as new content was written was the ongoing workflow.
blocker htmlproofer timing out on external link checks
Some external URLs (LinkedIn, GitHub) rate-limit automated HEAD requests, causing the workflow to fail intermittently on link validation. Fixed by adding --ignore-urls for known rate-limited domains and setting a timeout and retry configuration in the htmlproofer options.
resolved early Hugo build version mismatch between local and CI
Local Hugo version (0.128.0) differed from the version installed by apt in the Ubuntu runner, causing template rendering differences. Pinned the Hugo version in the workflow using peaceiris/actions-hugo with an explicit hugo-version to match local dev exactly.

05 What I Learned
→
Build your wordlist as you write, not after. Trying to add cspell exceptions retroactively across all existing content at once is painful. Committing cspell.json updates alongside new content keeps the list accurate and manageable.
→
Pin tool versions in CI from day one. The Hugo version mismatch was a reminder that "latest" in a CI runner and "latest" on your machine are rarely the same thing. Explicit version pinning across every tool in the workflow eliminates an entire class of intermittent failures.
→
External link checking needs a retry strategy, not just a timeout. Rate limits on external URLs aren't errors — they're expected behavior. Designing the validation step to retry with backoff (rather than hard-fail on the first timeout) makes the pipeline significantly more reliable without sacrificing coverage.
→
The pipeline is the product too. Treating this site as a real project — with branch protection, automated gates, and zero-touch deploys — produced better habits and better content quality than any manual review would have.
